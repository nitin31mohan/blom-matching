## Goal
Define the Attendee data schema and implement a synthetic data generator that produces realistic fake attendee profiles for development and testing.

## Purpose
Every downstream component — feature engineering, matching, the agent layer, and both frontend prongs — depends on a stable Attendee schema and a reliable source of realistic test data. This plan establishes that foundation. The demo prong also depends on synthetic data at runtime (no real PII ever).

## AC-1: Attendee schema is well-defined and validatable
```gherkin
Given a raw quiz response JSON object
When validated against data/schemas/quiz_response.json
Then valid attendees pass and invalid inputs (missing required fields, wrong types) are rejected
```

## AC-2: Synthetic generator produces deterministic output
```gherkin
Given a call to generate_attendees(n=50, seed=42)
When called twice with the same seed
Then both calls return identical Attendee lists (deterministic)
```

## AC-3: Generated data covers the full schema
```gherkin
Given generate_attendees(n=100, seed=0)
When the result is inspected
Then every Attendee has all required fields populated with plausible values,
  interests list has 2–6 items,
  energy_level is in range [1, 5],
  known_attendee_ids contains only valid IDs from the same batch
```

## AC-4: Known attendee pairs are symmetric
```gherkin
Given a generated batch of attendees
When attendee A lists attendee B in known_attendee_ids
Then attendee B also lists attendee A in known_attendee_ids
```

## AC-5: Corpus scales without error
```gherkin
Given a call to generate_attendees(n=500, seed=1)
When executed
Then it completes without error and returns exactly 500 Attendee objects
```

    Avoid: adding fields not listed above (schema must stay minimal for v1).

<task type="auto">
  <name>Task 2: Implement Attendee model and synthetic generator</name>
  <files>src/data/synthetic.py, src/data/__init__.py</files>
  <action>
    In src/data/synthetic.py:

    1. Define a Pydantic v2 BaseModel `Attendee` with fields matching quiz_response.json exactly. Use Python enums for enum fields. Add `model_config = ConfigDict(frozen=True)`.

    2. Implement `generate_attendees(n: int, seed: int = 0) -> list[Attendee]`:
       - Use `faker.Faker` seeded with `seed` for names and IDs
       - Use `random.Random(seed)` for all random choices (not the global random — must be reproducible)
       - Generate IDs as UUID4 strings first, then assign profiles
       - For known_attendee_ids: randomly pair ~20% of attendees as mutual friends (symmetric). Iterate through the list; for each attendee, randomly select 0–2 others to know, then add the reverse link. No self-references.
       - interests pool: at least 20 options (e.g., "music", "tech", "cooking", "fitness", "travel", "art", "gaming", "reading", "film", "sustainability", "entrepreneurship", "design", "photography", "sports", "science", "food", "fashion", "volunteering", "finance", "language-learning")
       - conversation_topics pool: at least 15 options (e.g., "current events", "career", "travel stories", "hobbies", "pop culture", "local scene", "tech trends", "health", "relationships", "personal growth", "food", "sports", "creativity", "books", "startups")

    3. In src/data/__init__.py:
       ```python
       from .synthetic import Attendee, generate_attendees
       __all__ = ["Attendee", "generate_attendees"]
       ```

    Avoid: using Python's global `random` module directly (breaks reproducibility). Always use the seeded `Random` instance.
  </action>
  <verify>python -c "from src.data import generate_attendees; a=generate_attendees(10,42); b=generate_attendees(10,42); assert a==b, 'Not deterministic'; print('OK', len(a))"</verify>
  <done>AC-2, AC-3, AC-4, AC-5 satisfied: deterministic, full schema, symmetric pairs, scales to 500</done>
</task>
