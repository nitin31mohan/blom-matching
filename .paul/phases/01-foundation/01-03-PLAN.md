<objective>
## Goal
Build the feature engineering pipeline: encode raw quiz responses into a weighted, L2-normalised feature vector ready for cosine similarity; compute Big Five proxy scores for the LLM explanation layer; apply event-type modifiers; handle missing data via event-level median imputation.

## Purpose
This is the algorithmic heart of Phase 01. The vector produced here is the direct input to Phase 02's cosine similarity and group assignment. Getting encoding, weighting, and imputation right here means Phase 02 can focus entirely on matching logic.

## Output
- `config/matching.yaml` — runtime config (sensitive_field_mode, weights)
- `src/features/weights.py` — weight config dict + dimension group definitions
- `src/features/encoder.py` — `build_feature_vector(user_quiz, event_id, all_users_quiz) -> UserFeatureVector`
- `src/features/modifiers.py` — `apply_event_modifiers(vectors, event_type) -> list[UserFeatureVector]`
- `src/features/__init__.py` — public exports
- `tests/features/test_encoder.py`, `test_weights.py`, `test_modifiers.py` — unit tests

## Deferred issue resolution
The SKILL says "Likert items (13 fields)" but the schema has 12 int 1-5 fields.
Resolution: 12 true Likert fields (min-max scaled) + `weekend_energy_level` counted as
the 13th (ordinal, [0,1] scaled identically). `preferred_activity_time` is the 14th
ordinal. Both ordinals are treated as continuous for similarity, same as Likert.
Total continuous dimensions in neutral mode: 14 (12 Likert + 2 ordinal).
</objective>

## AC-1: Encoding correctness
```gherkin
Given a raw quiz response dict with all 20 fields present
When build_feature_vector() is called
Then all 12 Likert values in the output vector are scaled to [0.0, 1.0] via (raw-1)/4
And both ordinal values (weekend_energy_level, preferred_activity_time) are scaled to [0.0, 1.0]
And nominal categoricals (industry, conversation_style, humour_style) are one-hot encoded
And the vector is L2-normalised (numpy.linalg.norm ≈ 1.0)
```

```gherkin
Given two users with identical profiles
  except user_a has energised_meeting_people=5
  and   user_b has energised_meeting_people=1

Then the social energy dimension contributes proportionally more to their distance
  than an equivalent difference on a 1.0x weight field
```

And the flag "low_profile_confidence" is present in flags

    **build_feature_vector():**
    ```python
    def build_feature_vector(
        user_quiz: dict,          # The user's quiz_responses dict (20 fields)
        user_id: str,             # pipeline_user_id
        event_id: str,
        all_users_quiz: list[dict],  # All users in same event (for median imputation)
        config: dict | None = None,  # Pass explicit config for testing; None = use _CONFIG
    ) -> UserFeatureVector:
    ```

    Implementation steps:
    1. **Impute missing fields**: for each quiz field with None value, compute
       event-level median/mode across all_users_quiz. Track in imputed_fields list.
       Emit "low_profile_confidence" flag if >= 3 fields were imputed.

    2. **Encode each field** to raw_encoded dict:
       - Likert (12 fields): `(raw - 1) / 4` → float in [0, 1]
         Emit "high_anxiety" flag if anxious_in_social_situations raw value >= 4.
       - Ordinal (weekend_energy_level, preferred_activity_time): lookup in ORDINAL_ENCODINGS
       - Binary (gender): man=0.0, woman=1.0
       - Nominal (industry, conversation_style, humour_style): one-hot list
       - Sensitive (country, religious_identity): one-hot if mode != "neutral",
         else zero vector of correct length

    3. **Build weight multiplier map** from DIMENSION_GROUPS. For sensitive fields,
       override weight to 0.0 if mode == "neutral", or the group's default if affinity/diversity.
       For "diversity" mode: encode sensitive fields as one-hot but NEGATE the similarity
       contribution (handled at Phase 02 similarity level — just encode normally here).

    4. **Assemble weighted vector**: concatenate all encoded dimensions, multiply each
       dimension by its field's weight. Maintain a stable field order (deterministic).

    5. **L2-normalise**: `vector = vector / numpy.linalg.norm(vector)`.
       If norm is 0 (all-zero vector, e.g. all fields missing): return zero vector.

    6. **Compute Big Five proxies** (stored in big_five, NOT included in vector):
       - Extraversion: mean of scaled(energised_meeting_people, comfortable_knowing_nobody)
       - Neuroticism: scaled(anxious_in_social_situations), reported as "Emotional stability" = 1 - value
       - Openness: mean of scaled(enjoys_unfamiliar_experiences, interested_in_current_events, eco_friendly_choices)
       - Conscientiousness: scaled(shows_up_on_time)
       - Agreeableness: mean of scaled(keeps_atmosphere_harmonious, messages_regularly_after_clicking)

    Avoid: z-scoring any field; including gender in the similarity vector;
    modifying all_users_quiz in place; caching state between calls.

    ```python
    def apply_event_modifiers(
        vectors: list[UserFeatureVector],
        event_type: str,          # "singles" | "social"
        user_genders: dict[str, str],  # {user_id: "man"/"woman"/"unknown"}
        config: dict | None = None,
    ) -> list[UserFeatureVector]:
    ```

    For each UserFeatureVector, produce a new (frozen) UserFeatureVector with:

    **singles events:**
    - Re-weight social energy dimensions by additional 1.3x (multiply the already-weighted
      social energy dimensions in the vector by 1.3, then re-normalise the entire vector).
    - Emit "gender_imbalance" flag on any user whose group has >75% one gender.
      Since groups aren't assigned yet at encoding time, flag is added if the WHOLE
      event corpus (the vectors list passed in) has >75% one gender — this signals
      a corpus-level imbalance for the operator to review.
      Note: per-group imbalance flagging happens in Phase 02 (assignment module).

    **social events:**
    - Re-weight values alignment dimensions by additional 1.2x, then re-normalise.

    Re-weighting mechanism: identify dimensions belonging to the target group by
    inspecting a dimension index map (build this from DIMENSION_GROUPS field order
    and NOMINAL_OPTIONS expansion). Multiply those dimensions by the extra factor,
    then L2-normalise the full vector again.

    Avoid: modifying the input vectors list; building the dimension index map by
    guessing — derive it from the same ordering used in encoder.py.

    **Add to encoder.py** — expose a module-level helper:
    ```python
    def get_dimension_index_map(config: dict | None = None) -> dict[str, list[int]]:
        """Return {field_name: [indices]} for each field in the vector."""
    ```
    modifiers.py imports and uses this to locate group dimensions.
